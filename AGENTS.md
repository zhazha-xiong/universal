# 全局开发规范

本文档适用于本仓库后续所有功能开发，不仅限于 MCP 网关。

## Git 工作流

- 使用 Git 管理代码。
- `docs/` 目录仅用于本地设计和讨论记录，不上传到仓库。
- 功能开发统一从 `main` 新建 `feature/...` 分支，严禁直接在 `main` 开发。
- 每个 feature 分支只做一个清晰目标，避免把多个功能混在同一个 PR 中。
- 开发完一个功能后提交 PR，等待审批合并。
- PR 合并后，先拉取最新 `main`，再基于最新 `main` 新建下一个 feature 分支。
- 允许执行 `commit` 和 `push`。
- commit message 使用中文，保持言简意赅，不携带任何额外 AI 标识。
- monorepo 下 commit message 需要带 scope，例如：`feat(mcp_gateway): 初始化网关工程骨架`。

## 开发原则

- 严格遵循 Go 语言规范和惯用写法。
- 尽量按已确认的项目约定开发。
- 如果实际开发情况与文档约定不一致，不要盲目继续，应先提出问题讨论。
- 如无必要，不新增实体或抽象。
- 长函数可以按步骤拆成多个小函数，但不要机械拆分。
- 每个功能都需要配套单元测试。

## 命名规范

- 函数名、结构体名、变量名不要用包名开头。
- 例如 `write` 包下的 `time` 变量不应命名为 `writeTime`，直接命名为 `time` 即可。
- 非导出辅助函数名需要简洁且能表达含义。
- 辅助函数名不要带 `helper`。
- 如果非导出辅助函数仅靠名称无法看出含义，需要补充 Go 风格函数注释。

## 注释规范

- 导出函数、导出类型需要添加必要的 Go 风格注释。
- 注释应说明用途、行为或关键约束，不写空泛描述。
- 非导出函数默认不要求注释；当函数名无法完整表达含义或逻辑存在关键约束时，需要添加 Go 风格注释。

## 测试规范

- 新功能必须写单元测试。
- MVP 阶段测试优先覆盖对外接口和关键业务逻辑。
- 测试应覆盖成功路径、关键参数错误和核心异常路径。
- 不为了覆盖率盲目测试私有实现细节。

## 配置与文档

- 配置文件优先使用 YAML。
- 关键信息使用 `${}` 占位，服务启动时从环境变量注入。
- 数据库、Redis、Token 等敏感配置不写死在仓库中。
- 项目设计文档默认放在本地 `docs/` 目录，不纳入 Git。
